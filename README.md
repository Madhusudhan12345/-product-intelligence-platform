# Autonomous Product Intelligence & Decision Support System

A multi-agent RAG platform that answers business-critical product/customer/engineering
questions by retrieving, reasoning across, and citing evidence from heterogeneous
organizational data sources (support tickets, PRDs, meeting notes, GitHub issues,
customer interviews, release notes).

## Why this scope

The original brief calls for a fully general production platform (advanced RAG,
multi-agent workflows, long-term memory, deep research, evaluation/observability,
workflow automation, production deployment). Building all of that at unlimited depth
isn't realistic in an internship timeline, so this implementation makes a deliberate
scoping decision: **go deep on a smaller number of pieces rather than shallow on all
of them.** Specifically:

- 3 agents (not a large swarm) with a clear, inspectable responsibility split
- 1 well-executed deep-research mode (multi-hop retrieval) rather than several shallow modes
- A real (if simple) evaluation harness with actual precision/recall numbers, not just a claim of "we have evals"
- A working long-term memory loop, not just a vector store being called "memory"

## Architecture

```
                     ┌─────────────────────┐
   Source Docs  ───► │  Ingestion Service   │  (chunking: 500 chars, 50 overlap)
 (tickets, PRDs,     └──────────┬───────────┘
  issues, notes,                │
  interviews)                   ▼
                     ┌─────────────────────┐
                     │   FAISS Vector Store │  (all-MiniLM-L6-v2 embeddings,
                     │  + metadata sidecar  │   cosine similarity, source_type/date filters)
                     └──────────┬───────────┘
                                │
   User Question                ▼
        │            ┌─────────────────────┐
        │            │   Memory Store        │ ◄── keyword-matched prior insights
        │            │   (SQLite)            │      logged from past queries
        │            └──────────┬───────────┘
        ▼                       │
┌───────────────────────────────▼──────────────────────────┐
│                      ORCHESTRATOR                          │
│                                                              │
│  1. Retriever Agent   -- hybrid vector + metadata search    │
│  2. Analyst Agent     -- reasons over evidence, cites        │
│                          chunk_ids, incorporates memory       │
│  3. [deep_research]   -- planner decides if a follow-up      │
│     mode only            search would close an evidence gap  │
│                          (up to 2 hops)                        │
│  4. Report Agent      -- reformats analysis into the right    │
│                          structure (direct answer / ranked     │
│                          list / exec summary)                  │
│                                                              │
└───────────────────────────────┬──────────────────────────┘
                                 │
                                 ▼
                     Answer + Evidence + Agent Trace
                     (logged back into Memory Store)
```

### Why FAISS + sentence-transformers over a managed vector DB
Same embedding stack used in prior projects (all-MiniLM-L6-v2, 384-dim). For the
data volumes in a system like this (thousands to low tens-of-thousands of chunks per
organization), a flat FAISS index with metadata filtering is simpler to reason about
and cheaper to run than standing up a managed vector DB, and it's trivial to swap
later if scale demands it (the `VectorStore` class is the only place that would change).

### Why 3 agents instead of many
Retriever / Analyst / Report maps directly onto the three distinct jobs a human analyst
does: find the data, reason about it, present it. Splitting further (e.g. separate
"trend detection agent", "risk agent") would add orchestration complexity without a
corresponding gain in answer quality for this evidence base size — a classic
over-engineering trap for take-home tasks.

### Deep research mode
For open-ended questions (e.g. "generate an executive summary of risks and
opportunities"), a lightweight **planner** step looks at the analyst's first-pass answer
and decides whether a follow-up search would close a gap. This is genuinely multi-hop
(not just "run RAG twice") — the follow-up query is generated by the model based on
what's missing from the first answer, capped at 2 hops to bound latency/cost.

### Long-term memory
Every query is logged (question, answer, agent trace, evidence source_ids). A distilled
"insight" (first ~300 chars of the analysis) is also stored separately. On future
queries, relevant prior insights are keyword-matched and passed to the Analyst Agent as
context — the model is instructed to re-verify rather than blindly trust them, so stale
or wrong past conclusions don't silently propagate.

*Known limitation:* the memory recall step uses keyword matching, not embedding
similarity. This is called out explicitly rather than glossed over — it was a deliberate
time trade-off. Swapping in the same FAISS store (querying a `memory` index instead of
keyword LIKE) is the natural next step and is noted in Future Improvements.

### Evaluation & observability
`backend/eval/` contains a small hand-labeled eval set (5 questions with the source
documents that *should* be retrieved) and a runner that computes retrieval precision
and recall against the live system, plus latency. This is intentionally small — the
point is a working, honest measurement loop, not a large benchmark suite.

Every query's full agent trace (which agent did what, in what order, referencing which
chunk IDs) is returned in the API response and persisted, giving the reviewer/PM
visibility into *why* the system answered the way it did — this is the "explainable,
evidence-backed answers" requirement made concrete.

## Project structure

```
backend/
  app/
    agents/          retriever, analyst, report agents + orchestrator
    models/           pydantic schemas
    routers/          FastAPI routes (ingest, query)
    services/         vectorstore (FAISS), memory (SQLite), ingestion, llm_client
    config.py
    main.py
  data/
    sample_sources.json   mock support tickets / PRDs / issues / notes / interviews
  eval/
    eval_set.json
    run_eval.py
  seed.py             one-off script to load sample data into FAISS
  requirements.txt
  .env.example
frontend/
  src/App.jsx         chat UI: mode toggle (standard/deep research), source filters,
                       answer + evidence panel + live agent trace
```

## Running locally

**Backend**
```bash
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # add your ANTHROPIC_API_KEY
python seed.py          # loads sample_sources.json into FAISS
uvicorn app.main:app --reload --port 8000
```

**Frontend**
```bash
cd frontend
npm install
npm run dev
```

**Run eval**
```bash
cd backend
python -m eval.run_eval
```

at the Render backend URL via `VITE_API_BASE`.

## Future improvements
- Replace keyword-based memory recall with embedding similarity against a dedicated memory index
- Add a re-ranker (e.g. cross-encoder) on top of FAISS top-k for higher precision
- Structured entity extraction at ingestion time (customer names, feature names) to enable exact filtering, not just semantic search
- Streaming responses from the agents instead of blocking until the full trace completes
- Swap flat FAISS index for HNSW or a managed vector DB once chunk volume grows past ~100k
- Add authentication/RBAC in front of the API before any real deployment
